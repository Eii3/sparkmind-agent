# 阿里巴巴 Java 开发手册 终极版 (1.3.0)

## 前言

《阿里巴巴 Java 开发手册》是阿里巴巴集团技术团队集体智慧与实战经验的结晶，经大规模一线实战检验并持续完善。手册以 Java 开发者为核心，从**编程规约、异常日志、单元测试、安全规约、工程结构、MySQL 数据库**六大维度规范开发，按约束力和故障敏感性分为**强制、推荐、参考**三类，配套**说明、正例、反例**辅助理解，愿景是实现高效协作、提升代码质量，减少开发踩坑。

本版本为 1.3.0 PDF 终极版，2017 年 10 月 14 日杭州云栖大会首发配套代码规约插件，阿里云效也集成了代码规约扫描引擎。

## 一、编程规约

### (一) 命名风格

1. 【强制】命名不能以`_`/`$`开头或结尾。

2. 【强制】禁止拼音与英文混合 / 直接用中文命名，国际通用拼音（如 alibaba）视同英文。

3. 【强制】类名用`UpperCamelCase`，DO/BO/DTO/VO/AO 除外。

4. 【强制】方法名、参数名、成员变量、局部变量用`lowerCamelCase`。

5. 【强制】常量全大写，单词间用`_`分隔，语义完整。

6. 【强制】抽象类以`Abstract/Base`开头，异常类以`Exception`结尾，测试类以 “被测试类名 + Test” 结尾。

7. 【强制】数组定义为`String[] args`，禁止`String args[]`。

8. 【强制】POJO 中布尔类型变量不加`is`，避免框架序列化解析错误。

9. 【强制】包名全小写，点分隔符间仅一个自然语义英文单词，包名用单数，类名有复数含义可使用复数。

10. 【强制】杜绝无规范缩写，避免望文不知义。

11. 【推荐】命名用完整单词组合实现代码自解释。

12. 【推荐】使用设计模式的模块 / 接口 / 类 / 方法，命名体现具体模式（如`OrderFactory`）。

13. 【推荐】接口方法 / 属性不加修饰符（含 public），仅定义应用基础常量，添加有效 Javadoc 注释。

14. 【强制 / 推荐】Service/DAO 类：接口暴露，实现类加`Impl`后缀；能力型接口用形容词（-able 形式）命名。

15. 【参考】枚举类名加`Enum`后缀，枚举成员全大写，单词间用`_`分隔。

16. 【参考】各层命名：

    - Service/DAO：`get`(单对象)/`list`(多对象)/`count`(统计)/`save/insert`(插入)/`remove/delete`(删除)/`update`(修改)；
    - 领域模型：`xxxDO`(数据对象)/`xxxDTO`(传输对象)/`xxxVO`(展示对象)，POJO 是统称，禁止命名`xxxPOJO`。



### (二) 常量定义

1. 【强制】禁止魔法值直接出现在代码中。
2. 【强制】long/Long 初始化用大写`L`，避免与数字`1`混淆。
3. 【推荐】按功能归类维护常量，避免单一大类常量类。
4. 【推荐】常量复用分五层：跨应用共享（二方库）、应用内共享（一方库）、子工程内共享、包内共享、类内共享。
5. 【推荐】值在固定范围且带延伸属性的变量，定义为枚举类。

### (三) 代码格式

1. 【强制】大括号：空代码块`{}`；非空块左大括号前不换行、后换行，右大括号前换行，后有 else 则不换行、终止则换行。
2. 【强制】左右小括号与字符间无空格。
3. 【强制】if/for/while/switch/do 与括号间加空格。
4. 【强制】二目 / 三目运算符左右各加一个空格。
5. 【强制】4 个空格缩进，禁止 tab（若用 tab 需设为 4 个空格）。
6. 【强制】注释双斜线与内容间仅一个空格。
7. 【强制】单行字符≤120，超出行按规则换行（第二行缩进 4 空格、运算符 / 点符号随下文换行等）。
8. 【强制】方法参数定义 / 传入时，逗号后加空格。
9. 【强制】IDE 编码设为 UTF-8，换行符用 Unix 格式。
10. 【推荐】不刻意用空格对齐行内字符，避免维护累赘。
11. 【推荐】方法内不同业务 / 语义逻辑间加一个空行，相同逻辑不加。

### (四) OOP 规约

1. 【强制】直接用类名访问静态变量 / 方法，避免通过对象引用。
2. 【强制】覆写方法必须加`@Override`注解。
3. 【强制】可变参数仅用于相同参数类型 / 业务含义，放参数列表最后，尽量避免使用。
4. 【强制】外部调用 / 二方库依赖的接口，禁止修改方法签名，过时加`@Deprecated`并说明替代方案。
5. 【强制】禁止使用过时的类 / 方法。
6. 【强制】用常量 / 确定有值的对象调用`equals`，避免 NPE（推荐`java.util.Objects#equals`）。
7. 【强制】包装类对象值比较一律用`equals`，避免 Integer 缓存区间坑。
8. 【强制 / 推荐】POJO/RPC 方法参数 / 返回值用包装类型，局部变量用基本类型。
9. 【强制】DO/DTO/VO 等 POJO 类属性不设默认值。
10. 【强制】序列化类新增属性不修改`serialVersionUID`，完全不兼容升级时修改。
11. 【强制】构造方法禁止加业务逻辑，初始化逻辑放`init`方法。
12. 【强制】POJO 类必须写`toString`方法，继承父类需加`super.toString`。
13. 【推荐】访问`String.split`结果数组时，检查最后一个分隔符后是否有内容，避免数组越界。
14. 【推荐】多个构造方法 / 同名方法按顺序放置。
15. 【推荐】类内方法顺序：公有 / 保护方法 > 私有方法 > getter/setter 方法。
16. 【推荐】setter 方法参数名与成员变量一致，getter/setter 不加业务逻辑。
17. 【推荐】循环体内字符串连接用`StringBuilder.append`。
18. 【推荐】非继承类、不可变引用域对象、不可重写方法、不可重赋值局部变量用`final`。
19. 【推荐】慎用`Object.clone`，默认浅拷贝，深拷贝需重写。
20. 【推荐】类成员与方法访问控制从严（如工具类构造方法私有、非 static 成员变量按作用域设 protected/private 等）。

### (五) 集合处理

1. 【强制】重写`equals`必须重写`hashCode`，Set 存储对象 / 自定义对象作为 Map 键时，必须重写两者。
2. 【强制】`ArrayList.subList`结果不能强转成 ArrayList，会抛`ClassCastException`。
3. 【强制】`subList`场景中，修改原集合元素个数会导致子列表操作抛`ConcurrentModificationException`。
4. 【强制】集合转数组用`toArray(T[] array)`，数组大小设为`list.size()`。
5. 【强制】`Arrays.asList()`转的集合，禁止调用 add/remove/clear，会抛`UnsupportedOperationException`。
6. 【强制】`<? extends T>`集合不能 add，`<? super T>`集合不能 get，遵循 PECS 原则（读用 extends、写用 super）。
7. 【强制】禁止在 foreach 循环中增删元素，增删用 Iterator，并发操作需加锁。
8. 【强制】JDK7 + 中，Comparator 需满足自反性、传递性、对称性，否则排序抛`IllegalArgumentException`。
9. 【推荐】集合初始化时指定初始大小（如 HashMap：`(元素个数/0.75)+1`）。
10. 【推荐】遍历 Map 用`entrySet`，效率高于`keySet`（JDK8 推荐`Map.foreach`）。
11. 【推荐】注意 Map 集合 K/V 能否存 null（如 Hashtable/ConcurrentHashMap 均不允许，HashMap 均允许）。
12. 【参考】利用集合的有序性 / 稳定性，避免无序性 / 不稳定性的负面影响。
13. 【参考】用 Set 去重，效率高于 List 的 contains 方法。

### (六) 并发处理

1. 【强制】单例对象及其中方法保证线程安全。
2. 【强制】创建线程 / 线程池时指定有意义的线程名称，方便问题回溯。
3. 【强制】线程资源通过线程池提供，禁止自行显式创建线程。
4. 【强制】通过`ThreadPoolExecutor`创建线程池，禁止使用`Executors`（避免 OOM 风险）。
5. 【强制】`SimpleDateFormat`不定义为 static 变量，若定义需加锁或用`ThreadLocal`（JDK8 推荐`DateTimeFormatter`）。
6. 【强制】高并发时优先使用无锁数据结构，缩小锁粒度（锁区块 > 锁方法体，对象锁 > 类锁）。
7. 【强制】多资源加锁保持一致顺序，避免死锁。
8. 【强制】并发修改同一记录需加锁（应用层 / 缓存 / 数据库乐观锁，version 为依据，冲突率 < 20% 用乐观锁，重试≥3 次）。
9. 【强制】多线程定时任务用`ScheduledExecutorService`，替代 Timer（Timer 一个任务异常会导致其他任务终止）。
10. 【推荐】用`CountDownLatch`做异步转同步，子线程需 catch 异常并调用`countDown`。
11. 【推荐】多线程避免共享`Random`实例，JDK7 + 用`ThreadLocalRandom`。
12. 【推荐】双重检查锁实现延迟初始化，目标属性声明为`volatile`（JDK5+）。
13. 【参考】`volatile`解决一写多读的内存不可见问题，多写需用`AtomicInteger/LongAdder`（JDK8）。
14. 【参考】高并发下 HashMap resize 可能出现死链，需用其他数据结构或加锁。
15. 【参考】`ThreadLocal`对象建议用 static 修饰，无法解决共享对象更新问题。

### (七) 控制语句

1. 【强制】switch 块中每个 case 需用 break/return 终止或注释说明，必须包含 default 并放在最后。
2. 【强制】if/else/for/while/do 语句必须用大括号，禁止单行编码。
3. 【推荐】异常分支用卫语句替代 if-else，if-else-if 层数不超过 3 层（超层用策略模式 / 状态模式）。
4. 【推荐】条件判断中避免复杂语句，将结果赋值给有意义的布尔变量。
5. 【推荐】循环体中耗时操作（如对象定义、数据库连接）移至循环外。
6. 【推荐】批量操作接口做入参保护。
7. 【参考】需要参数校验的场景：低频次方法、高耗时方法、高稳定性方法、开放接口、敏感权限入口。
8. 【参考】无需参数校验的场景：循环调用方法、底层高频方法、私有方法（入参已校验）。

### (八) 注释规约

1. 【强制】类、类属性、类方法用 Javadoc 规范（`/** 内容 */`），禁止`//`。
2. 【强制】抽象方法 / 接口方法的 Javadoc，需说明方法功能、返回值、参数、异常及子类实现要求。
3. 【强制】所有类添加创建者和创建日期。
4. 【强制】方法内部单行注释在被注释语句上方另起一行（`//`），多行注释用`/* */`并与代码对齐。
5. 【强制】枚举类型字段必须加注释说明用途。
6. 【推荐】用中文注释清晰说明问题，专有名词 / 关键字保留英文。
7. 【推荐】代码修改时同步更新注释（参数、返回值、核心逻辑等）。
8. 【参考】谨慎注释代码，无用则删除，需保留则详细说明动机。
9. 【参考】注释准确反应设计思想和业务含义，兼顾可读性和维护性。
10. 【参考】好的命名和代码结构实现自解释，注释精简准确，避免冗余。
11. 【参考】特殊注释标记注明人 / 时间，及时处理：`TODO`（待实现）、`FIXME`（错误需修复）。

### (九) 其它

1. 【强制】正则表达式利用预编译功能（`Pattern.compile`放方法体外）。
2. 【强制】velocity 调用 POJO 属性直接用属性名，框架自动调用 getXxx/isXxx。
3. 【强制】后台给页面的变量用`$!{var}`，避免 null 显示为字符串。
4. 【强制】获取整数随机数用`Random.nextInt/nextLong`，避免`Math.random()`放大取整。
5. 【强制】获取当前毫秒数用`System.currentTimeMillis()`，替代`new Date().getTime()`（JDK8 推荐`Instant`）。
6. 【推荐】视图模板中不加入复杂逻辑，遵循 MVC 原则。
7. 【推荐】数据结构构造 / 初始化指定大小，避免内存耗尽。
8. 【推荐】及时清理无用代码段 / 配置信息，暂存代码用`///`注释理由。

## 二、异常日志

### (一) 异常处理

1. 【强制】RuntimeException（如 NPE、数组越界）通过预检查规避，不通过 catch 处理（外部输入解析等无法预检查的除外）。
2. 【强制】不使用异常做流程 / 条件控制，处理效率低于条件分支。
3. 【强制】区分稳定 / 非稳定代码，非稳定代码 catch 按异常类型处理，避免大段代码统一 try-catch。
4. 【强制】捕获异常必须处理，不处理则向上抛出，最外层业务使用者需转化为用户可理解的内容。
5. 【强制】事务代码中的 try 块，catch 后需手动回滚事务（若需要）。
6. 【强制】finally 块必须关闭资源 / 流，有异常需嵌套 try-catch（JDK7 + 用 try-with-resources）。
7. 【强制】finally 块中禁止使用 return，会覆盖 try 块的 return 结果。
8. 【强制】捕获异常需与抛出异常完全匹配或为其父类。
9. 【推荐】方法返回 null 需添加注释说明场景，调用方做 null 判断。
10. 【推荐】注意 NPE 产生场景（自动拆箱、数据库查询 null、级联调用等），JDK8 用`Optional`规避。
11. 【推荐】自定义有业务含义的异常，避免直接抛出`RuntimeException/Exception/Throwable`。
12. 【参考】开放接口用错误码，应用内部用异常抛出，跨应用 RPC 调用用 Result 封装（isSuccess + 错误码 + 信息）。
13. 【参考】遵循 DRY 原则，抽取共性异常处理逻辑，避免代码重复。

### (二) 日志规约

1. 【强制】依赖 SLF4JAPI 记录日志，禁止直接使用 Log4j/Logback 原生 API。
2. 【强制】日志文件至少保存 15 天，应对周频次异常。
3. 【强制】扩展日志命名：`appName_logType_logName.log`（logType：stats/desc/monitor/visit 等）。
4. 【强制】trace/debug/info 级日志用条件输出或占位符，避免字符串拼接浪费资源。
5. 【强制】log4j.xml 中设置`additivity=false`，避免重复打印日志。
6. 【强制】异常日志包含案发现场信息和堆栈信息，不处理则向上抛出。
7. 【推荐】生产环境禁止输出 debug 日志，谨慎输出 info/warn 日志，避免磁盘占满。
8. 【参考】用户输入参数错误用 warn 日志，系统逻辑错误用 error 日志。

## 三、单元测试

1. 【强制】单元测试遵守**AIR 原则**：Automatic（自动化）、Independent（独立性，用例互不调用）、Repeatable（可重复，不受外部环境影响）。
2. 【强制】单元测试全自动执行，用`assert`验证结果，禁止`System.out`人肉验证。
3. 【强制】测试粒度至方法级别，精确定位问题，不负责跨类 / 跨系统交互逻辑。
4. 【强制】核心业务 / 应用 / 模块的增量代码，必须通过单元测试。
5. 【强制】单元测试代码放在`src/test/java`，禁止放在业务代码目录。
6. 【推荐】单元测试语句覆盖率≥70%，核心模块语句 / 分支覆盖率≥100%。
7. 【推荐】编写单元测试遵守**BCDE 原则**：Border（边界值测试）、Correct（正确输入测试）、Design（结合设计文档）、Error（错误输入测试）。
8. 【推荐】数据库相关测试，通过程序插入数据，设定自动回滚或给数据加专属前缀，避免脏数据。
9. 【推荐】对不可测代码进行重构，避免编写不规范测试代码。
10. 【推荐】设计评审阶段确定单元测试范围，覆盖所有测试用例，提测前完成编写。
11. 【参考】业务代码避免：构造方法逻辑过多、全局 / 静态方法过多、外部依赖过多、条件语句层数过多。
12. 【参考】纠正单元测试误解：非测试同学职责、代码多余、无需维护、与线上故障无关。

## 四、安全规约

1. 【强制】用户个人页面 / 功能必须做权限控制校验，防止水平越权。
2. 【强制】用户敏感数据展示前必须脱敏（如手机号 158****9119）。
3. 【强制】SQL 参数使用参数绑定 / METADATA 字段限定，禁止字符串拼接，防止 SQL 注入。
4. 【强制】用户请求参数必须做有效性验证，防止内存溢出、慢查询、重定向、注入、ReDoS 等问题。
5. 【强制】向 HTML 页面输出用户数据前，做安全过滤 / 正确转义。
6. 【强制】表单 / AJAX 提交必须执行 CSRF 安全过滤，防止跨站请求伪造。
7. 【强制】短信 / 邮件 / 支付等平台资源使用，实现防重放限制（数量 / 频率 / 验证码），避免滥刷和资损。
8. 【推荐】发贴 / 评论 / 即时消息等用户生成内容场景，实现防刷和违禁词过滤。

## 五、MySQL 数据库

### (一) 建表规约

1. 【强制】是 / 否字段命名为`is_xxx`，类型为`unsigned tinyint`（1 = 是，0 = 否），非负数字段均用 unsigned。
2. 【强制】表名 / 字段名用小写字母 / 数字，禁止数字开头、两个下划线间仅数字（兼容 Linux 大小写敏感）。
3. 【强制】表名用单数名词，对应 DO 类名也为单数。
4. 【强制】禁用 MySQL 保留字（如 desc、range）。
5. 【强制】索引命名：主键`pk_字段名`、唯一索引`uk_字段名`、普通索引`idx_字段名`。
6. 【强制】小数类型用`decimal`，禁止`float/double`（避免精度损失）。
7. 【强制】字符串长度几乎相等时用`char`，可变长用`varchar`（长度≤5000，超 5000 用 text 并独立建表）。
8. 【强制】表必备三字段：`id`（unsigned bigint，主键自增）、`gmt_create`（datetime，创建时间）、`gmt_modified`（datetime，更新时间）。
9. 【推荐】表名命名为`业务名称_表的作用`，库名与应用名称一致。
10. 【推荐】修改字段含义 / 状态时，同步更新字段注释。
11. 【推荐】字段适当冗余（非频繁修改、非超长字段），提升查询性能，保证数据一致性。
12. 【推荐】单表行数 > 500 万或容量 > 2GB 时，进行分库分表。
13. 【参考】根据数据范围选择合适的字符 / 数值类型，节约空间并提升检索速度。

### (二) 索引规约

1. 【强制】业务唯一特性的字段 / 字段组合，必须建唯一索引（避免脏数据，提升查询速度）。
2. 【强制】关联查询表数≤3，被关联字段数据类型一致且建索引。
3. 【强制】varchar 字段建索引时指定长度，根据文本区分度确定（一般 20 个字符区分度≥90%）。
4. 【强制】页面搜索禁止左模糊 / 全模糊，需实现则用搜索引擎。
5. 【推荐】order by 场景利用索引有序性，排序字段放组合索引最后，避免 file_sort。
6. 【推荐】使用覆盖索引查询，避免回表（explain 的 extra 列显示`using index`）。
7. 【推荐】超多分页场景用延迟关联 / 子查询优化，避免大 offset 低效查询。
8. 【推荐】SQL 性能优化目标：至少`range`级别，推荐`ref`级别，最优`consts`级别。
9. 【推荐】组合索引按 “区分度高→低” 排序，等号条件字段前置（混合等号 / 非等号时）。
10. 【推荐】避免字段类型不同导致的隐式转换，造成索引失效。
11. 【参考】创建索引避免极端：宁滥勿缺、宁缺勿滥、抵制惟一索引。

### (三) SQL 语句

1. 【强制】用`count(*)`统计行数，替代`count(列名)/count(常量)`（SQL92 标准，与 NULL 无关）。
2. 【强制】`count(distinct col1,col2)`中若一列全为 NULL，结果为 0。
3. 【强制】使用`sum()`时需判空，避免 NPE（推荐`IF(ISNULL(SUM(g)),0,SUM(g))`）。
4. 【强制】用`ISNULL()`判断 NULL 值，避免直接比较（NULL 与任何值比较结果为 NULL）。
5. 【强制】分页查询时，若 count=0 直接返回，避免执行分页语句。
6. 【强制】禁止使用外键与级联，应用层实现关联逻辑（避免分布式 / 高并发下的阻塞和风暴）。
7. 【强制】禁止使用存储过程，难以调试、扩展和移植。
8. 【强制】删除 / 修改记录前先 select 验证，避免误操作。
9. 【推荐】`in`操作的集合元素数量控制在 1000 个之内。
10. 【参考】全球化场景用 utf-8 编码，存储表情用 utfmb4，区分`LENGTH`（字节数）和`CHARACTER_LENGTH`（字符数）。
11. 【参考】禁止开发代码中使用`TRUNCATE TABLE`（无事务、不触发 trigger，易造成事故）。

### (四) ORM 映射

1. 【强制】表查询明确指定字段列表，禁止使用`*`（减少解析成本，避免与 resultMap 不一致）。
2. 【强制】POJO 布尔属性不加`is`，数据库字段加`is_`，在 resultMap 中做映射。
3. 【强制】禁止用`resultClass`作为返回参数，必须定义 resultMap，实现字段与 DO 类解耦。
4. 【强制】sql.xml 中参数用`#{}`/`#param#`，禁止`${}`，防止 SQL 注入。
5. 【强制】不推荐使用 iBATIS 的`queryForList(String, int, int)`，避免全表查询后子集合截取。
6. 【强制】禁止用 HashMap/Hashtable 作为查询结果集输出，值类型不可控。
7. 【强制】更新记录时，同步更新`gmt_modified`为当前时间。
8. 【推荐】避免大而全的更新接口，仅更新有改动的字段（减少出错、提升效率、降低 binlog 存储）。
9. 【参考】慎用`@Transactional`，考虑回滚方案（缓存、搜索引擎、消息等），避免影响数据库 QPS。
10. 【参考】合理使用 ORM 标签：`<isEqual>`（常量对比）、`<isNotEmpty>`（非空非 null）、`<isNotNull>`（非 null）。

## 六、工程结构

### (一) 应用分层

**推荐分层（上层可依赖下层）**：终端显示层 → 开放接口层 → 请求处理层 (Web 层) → 业务逻辑层 (Service 层) → 通用处理层 (Manager 层) → 数据持久层 (DAO 层) → 数据源 / 外部接口 / 第三方平台

1. 开放接口层：封装 Service 为 RPC/HTTP 接口，做网关安全 / 流量控制；
2. 终端显示层：各类端的模板渲染（velocity/JS/JSP/ 移动端）；
3. Web 层：请求转发、基础参数校验、简单业务处理；
4. Service 层：具体业务逻辑实现；
5. Manager 层：第三方平台封装、Service 通用能力下沉、多 DAO 组合复用；
6. DAO 层：与底层数据库交互；
7. 外部接口 / 第三方平台：跨部门 RPC 接口、基础平台、外部 HTTP 接口。

**分层异常处理**：

- DAO 层：catch (Exception) 并抛 DAOException，不打印日志；
- Service 层：捕获异常并打印带参数的出错日志；
- Manager 层：同机部署则同 DAO 层，独立部署则同 Service 层；
- Web 层：不向上抛异常，跳转到友好错误页面并提示；
- 开放接口层：将异常转化为错误码 + 错误信息返回。

**分层领域模型**：

- DO：与数据库表一一对应，DAO 层传输；
- DTO：Service/Manager 层向外传输；
- BO：Service 层输出的业务逻辑封装对象；
- AO：Web/Service 层间的抽象复用对象，贴近展示层；
- VO：Web 层向模板渲染引擎传输的显示对象；
- Query：各层接收的查询请求对象，超 2 个参数封装，禁止用 Map。1

### (二) 二方库依赖

1. 【强制】GAV 命名规则：
    - GroupID：`com.{公司/BU}.业务线.[子业务线]`（最多 4 级）；
    - ArtifactID：`产品线名-模块名`（语义唯一，先查中央仓库）；
    - Version：`主版本号.次版本号.修订号`（主版本：架构 / API 大改；次版本：新增功能；修订号：修复 BUG，起始 1.0.0）。
2. 【强制】线上应用不依赖 SNAPSHOT 版本（安全包除外），保证发布幂等性。
3. 【强制】二方库新增 / 升级，保持非功能点 jar 包仲裁结果不变，差异点用`<excludes>`排除。
4. 【强制】二方库接口返回值，禁止使用枚举 / 包含枚举的 POJO。
5. 【强制】依赖二方库群时，定义统一版本变量，避免版本不一致。
6. 【强制】子项目 pom 中，禁止相同 GroupId+ArtifactId 对应不同 Version。
7. 【推荐】pom 中`<dependencies>`声明依赖，`<dependencyManagement>`做版本仲裁。
8. 【推荐】二方库尽量不增加配置项。
9. 【参考】二方库发布原则：精简可控（移除无用 API / 依赖，provided 引入第三方库，无具体日志实现）、稳定可追溯（记录版本变化，明确维护者和源码地址）。

### (三) 服务器

1. 【推荐】高并发服务器调小 TCP 的 time_wait 超时时间（linux：`net.ipv4.tcp_fin_timeout = 30`）。
2. 【推荐】调大服务器最大文件句柄数（fd），避免高并发下 “open too many files” 错误。
3. 【推荐】JVM 添加`-XX:+HeapDumpOnOutOfMemoryError`，OOM 时输出 dump 信息，便于问题排查。
4. 【推荐】生产环境 JVM 的 Xms 和 Xmx 设置为相同值，避免 GC 后堆大小调整的性能压力。
5. 【参考】服务器内部重定向用 forward，外部重定向用 URL 拼装工具类，避免 URL 维护不一致和安全风险。
